# P3.4 後端 API 整合評估

> 最後更新：2026-04-15
> 目的：評估是否、以及何時將 `grade-filter` 從純前端（IndexedDB）
> 改為連接 `api-server`（Express + Drizzle + PostgreSQL）。

---

## 1. 摘要（TL;DR）

**建議：暫不啟用後端整合。** 維持目前「純前端 + IndexedDB」架構。

- **目前架構優勢極強**：零伺服器成本、零資料外洩風險、零維運負擔、完全符合
  《個資法》避免爭議的設計（資料不離開老師電腦）。
- **啟用後端的實際收益有限**：唯一明確痛點是「跨裝置同步」，但對單一老師
  備課使用，影響程度不大。
- **若未來確有需求**，基礎建設已備（見 §3），從零建成可行系統估 2–4 週，
  但需額外處理身份認證、HTTPS 憑證、資料庫備份、個資合規審查。

**觸發重新評估的條件**（有任一為「是」時再評估啟用）：

- [ ] 單一學校有 3 位以上老師希望共享特殊生 / 優先名單清單
- [ ] 學校行政端要求「審計日誌」（誰、何時、篩選了哪些名單）
- [ ] 使用者在 2 台以上裝置間同步資料成為日常痛點
- [ ] 外部單位（縣市教育局）要求集中化統計

## 2. 後端可解鎖的能力

| 能力 | 目前狀態 | 啟用後端後 | 使用者受益程度 |
| --- | --- | --- | --- |
| 跨裝置同步 | ❌ 僅本機 | ✅ 雲端儲存 | ★★★ 對多裝置使用者有感 |
| 多人協作 | ❌ 單人 | ✅ 多帳號、角色權限 | ★★ 目前多為單師自用 |
| 資料集中管理 | ❌ 各老師獨立 | ✅ 學校統一 | ★★ 僅當校方有此需求 |
| 審計日誌 | ❌ 無 | ✅ 所有操作留痕 | ★★★ 對稽核敏感單位重要 |
| 離線使用 | ✅ 完全離線（P3.1 PWA） | ⚠ 需同步狀態管理 | **後端反而退步** |
| 資料備份 | ⚠ 依賴瀏覽器 | ✅ 伺服器自動備份 | ★★★ 消除誤清風險 |
| 個資外洩風險 | ✅ 零（僅在本機） | ⚠ 需 HTTPS + 加密 + DPO | **後端明顯退步** |

## 3. 既有基礎建設盤點

啟用後端不需從零開始，workspace 中已預留：

### 3.1 `artifacts/api-server/`

- **框架**：Express 5
- **中介層**：cors、pino-http（JSON 結構化日誌）、express.json / urlencoded
- **實作進度**：僅 `/api/health`（route/健康檢查），`src/routes/index.ts`
  為空架構。
- **缺口**：
  - 無身份認證（session / JWT / OAuth 均未接）
  - 無 rate limit、CSRF、helmet
  - 無 env 變數驗證（DATABASE_URL、PORT 等）
  - `routes/` 下只有 `health.ts`

### 3.2 `lib/db/`

- **ORM**：Drizzle（pg-core），搭配 drizzle-zod 產型別
- **實作進度**：`src/schema/index.ts` 僅有範例註解，**尚未定義任何表**
- **缺口**：
  - users 表
  - 成績資料（score_records）、特殊生（special_students）、
    優先名單（priority_students）、篩選設定（filter_configs）、
    篩選結果（filter_snapshots）皆未建
  - 無 Drizzle migration 產出機制

### 3.3 `lib/api-zod/` + `lib/api-client-react/`

- **定位**：OpenAPI / Zod schema 同步點 + React Query hooks 產生器
- **實作進度**：`src/generated/` 為空產物目錄，`custom-fetch.ts` 是
  base fetch wrapper 骨架
- **缺口**：需先有 OpenAPI spec（或 Zod schema 作為 source of truth）
  才能生成客戶端

### 3.4 `grade-filter` 當前資料流

```
File Upload → xlsx parse (Web Worker) → useState (AppContext)
                                            ↓
                                       IndexedDB (keyval-like, lib/storage.ts)
```

**改為 API 後**：

```
File Upload → xlsx parse → POST /api/scores/bulk → Postgres
Component → useQuery('/api/scores') ← ← Postgres
(離線時 fallback 回 IndexedDB，或顯示離線提示)
```

## 4. 真實啟用路徑（若要做）

### Phase A：身份認證（3–5 天）

1. 選 Better Auth / Lucia / Clerk 之一
2. `users` + `sessions` 表
3. 登入頁 / 註冊頁（grade-filter 新 `/login` 路由）
4. API middleware：`requireAuth` 保護所有 `/api/*` 除 `/health`、`/auth/*`

### Phase B：資料模型（2–3 天）

在 `lib/db/src/schema/` 定義：

```ts
// score_records — 單科成績（國文/英文/數學各一列）
export const scoreRecords = pgTable("score_records", {
  id: uuid("id").primaryKey().defaultRandom(),
  userId: uuid("user_id").references(() => users.id).notNull(),
  subject: text("subject").notNull(), // chinese | english | math
  grade: integer("grade").notNull(),
  className: text("class_name"),
  seatNo: text("seat_no"),
  studentName: text("student_name").notNull(),
  idNumber: text("id_number").notNull(), // ⚠ 需加密或 hash
  score: real("score"),
  importedAt: timestamp("imported_at").defaultNow(),
});

// special_students / priority_students / filter_configs / filter_snapshots
// …（略，架構類似）
```

**關鍵決策點**：身分證字號是否要以加密形式存放？建議至少以
`sha256(idNumber + user_salt)` 做查找鍵，顯示時取 `idNumber`
加密後存（例：libsodium secretbox）。

### Phase C：API 路由（5–7 天）

- `POST /api/scores/bulk` — 批次匯入
- `GET /api/scores?subject=&grade=` — 條件查詢
- `DELETE /api/scores?subject=` — 清除單科（對應目前 UI 的「清除」）
- 對應做 special-students / priority-students / filter-configs /
  filter-snapshots 五套 CRUD

### Phase D：前端整合（3–5 天）

1. 以 `@tanstack/react-query` 封裝 hooks（`lib/api-client-react` 已有殼）
2. `AppContext` 的 state 來源由 IndexedDB 切成 React Query cache
3. **保留 IndexedDB 作為離線 cache**（線上模式 = RQ，離線模式 = IDB）
4. 衝突解決策略：last-write-wins / 手動選擇（視需求）

### Phase E：部署（3–5 天）

- PostgreSQL：Neon / Supabase / 自架
- Server：Render / Railway / 自架 + Caddy
- HTTPS + 憑證自動更新
- 資料庫備份排程（至少每日，保留 30 天）
- 密碼輪替與 env 管理（`.env` + dotenv-vault / SOPS）

**總估工時**：約 16–25 天（合 2–4 週；與 7.4 節預估一致）

## 5. 風險與合規

啟用後端後新增的風險面：

### 5.1 個資法（臺灣）

- 身分證字號屬「個人資料保護法」第 2 條第 1 款之個人資料
- 若儲存於伺服器，學校需：
  - 指定個資保護聯絡人（校方行政）
  - 列入年度個資盤點
  - 擬定洩漏應變計畫
- 若以雲端服務（Neon/Supabase）儲存，需確認服務商是否在
  境內或有相當保護等級

### 5.2 可能的技術風險

| 風險 | 緩解措施 |
| --- | --- |
| SQL injection | Drizzle ORM 天然防護（不要 raw SQL） |
| XSS | React 預設轉義；勿使用 `dangerouslySetInnerHTML` |
| CSRF | SameSite=Strict cookie + CSRF token |
| 身分證外洩 | 靜態加密 + 欄位遮蔽 + access log |
| 暴力破解登入 | rate limit + 登入失敗延遲 |
| 備份外洩 | 備份檔同樣加密 |

### 5.3 對使用者信任的衝擊

目前「完全在本機運作」是本工具對老師的隱性信任優勢（相較 Google Form /
Excel 線上版）。一旦出現「伺服器」，即使再安全，老師第一反應會是
「資料是不是傳到雲端了？」，須在 UI 明確說明。

## 6. 建議的決策流程

若未來收到「是否要做後端」的提問，依此流程回答：

```
是否有至少 2 個「觸發條件」出現？（§1 的清單）
   │
   ├─ 否 → 維持現狀。IndexedDB + PWA 已覆蓋 95% 需求。
   │
   └─ 是 → 問：有無預算 / 人力承擔 2–4 週開發 + 持續維運？
              │
              ├─ 無 → 建議改用現成 SaaS（例：Google Sheets 共享）
              │      而非自建後端。
              │
              └─ 有 → 依 §4 Phase A → E 順序啟動。
                      每個 Phase 結束做一次 go / no-go 審查。
```

## 7. 過渡期混合策略（若有需要同步但不想全後端）

中間方案 A：**只同步設定**（不同步成績資料）

- 成績資料仍在 IndexedDB
- 僅把 `filterConfigs`、`filterTemplates`、`excelProfiles` 做雲端同步
- 工時：3–5 天（僅需一張 `user_settings` 表 + JSON blob 存）
- 隱私爭議小（不含學生個資）
- 覆蓋「換電腦繼續用相同篩選條件」的需求

中間方案 B：**匯出 / 匯入 JSON 備份**

- 完全不動後端
- 提供「匯出全部資料為 JSON 檔」「匯入備份」按鈕
- 工時：1 天
- 老師可用 Google Drive / OneDrive / USB 自行跨裝置搬移
- 對「資料安全」最友善

**兩者都比完整後端便宜且低風險**，可作為 P3.4 的降級實作。

---

## 8. 結論

| 情境 | 建議 |
| --- | --- |
| **預設（目前狀況）** | 維持純前端，等觸發條件發生 |
| **有跨裝置同步需求** | 優先實作中間方案 B（JSON 匯出入），再評估是否需要 A |
| **有學校 / 行政需求** | 啟動完整後端整合（§4 Phase A–E） |
| **完全沒有需求** | 保持簡單，IndexedDB + PWA 已是合適解答 |

P3.4 作為 **評估項目** 本次即以本文件結案；實作與否視未來使用者回饋
再決定。若下次改良規劃仍保留此項，建議先實作 §7 的中間方案 B
（工時最低、效益最即時、風險最小）。
